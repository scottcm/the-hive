# The Hive v1.1 — Deferred Items

Status: Draft
Created: 2026-03-07
Origin: V1_DESIGN.md architecture review (reviews/V1_UC_RECONCILIATION.md)

---

## Purpose

This document tracks features specified in V1_DESIGN.md that are deferred from
v1.0 implementation to v1.1. Each item was reviewed during the UC validation
process and determined to be additive — it can be introduced after v1.0 ships
without breaking the v1.0 contract.

V1_DESIGN.md contains inline markers (`<!-- v1.1: ... -->`) at each affected
section pointing back to this document.

---

## D1: Independent Task-Level Heartbeat

**Origin:** V1_DESIGN.md §7.9 (dual heartbeat contract), §8.6 (`heartbeat_deadline`)

**What v1.0 does instead:** Task claim liveness is derived from session liveness.
The recovery job releases all claims for an expired session. There is no
independent per-task heartbeat deadline — if the session is alive, all its
claimed tasks are alive. The `heartbeat_deadline` column exists in the schema
but is set equal to `lease_expires_at` at claim time and not independently
extended.

**What v1.1 adds:**
- `heartbeat_task` tool extends `heartbeat_deadline` independently of session lease.
- Recovery job gains a task-level sweep: release tasks where
  `heartbeat_deadline < now()` even if the session is still active.
- Agents must heartbeat both session and each active task independently.

**Why deferred:** The scenario where an agent has a live session but neglects a
specific task is real but edge-case. Session-derived liveness covers the common
failure mode (agent crash/disconnect). Independent task heartbeats add
operational complexity (agents must track per-task timers) without proportional
benefit until the system has data showing task-level neglect is a problem.

**Migration impact:** None. The `heartbeat_deadline` column already exists.
v1.1 changes application logic only — the recovery job gains a new query path,
and `heartbeat_task` becomes meaningful rather than a no-op alias.

## D2: Multi-Tier Review (Judgment + Coverage + Reconciliation)

**Origin:** V1_DESIGN.md §7.6, §7.4 (`task_type`), §8.6 (`task_type` CHECK)

**What v1.0 does instead:** A single `review` task type. One review child task
per implementation task. The reviewer performs a combined review (both judgment
and coverage concerns in a single pass). The `review_ref`, verdict lifecycle,
and G4 profile-level separation all apply unchanged.

**What v1.1 adds:**
- Split `review` into `review_judgment`, `review_coverage`, and
  `review_reconciliation` as distinct `task_type` values.
- Dual-review workflow: two review children + one reconciliation child per
  implementation task.
- Reconciliation task merges findings from both review passes.
- Parent implementation task cannot close until all review + reconciliation
  children are `done`.

**Why deferred:** The three-tier model means each implementation task spawns
2-3 review children, each needing a different agent, each with its own
claim/heartbeat/gate cycle. For a v1 system proving that agents can coordinate
at all, a single review task builds confidence faster. The three-tier model is
the target architecture but can be introduced once single-review coordination
is stable.

**Migration impact:** Low. Add new `task_type` enum values via constraint
replacement (same pattern as §8.11). Existing `review` tasks remain valid.
New review children use the split types. G0 child closure already handles
arbitrary child task types — no gate engine changes needed.

## D3: Clarification Auto-Escalation

**Origin:** V1_DESIGN.md §7.7 (escalation trigger)

**What v1.0 does instead:** Manual escalation only. The current routee calls
`update_clarification(id, routed_to=<next_in_chain>)` to pass a clarification
up the chain: milestone owner -> project owner -> `human-required`.

**What v1.1 adds:**
- Timeout-based auto-escalation via the background recovery job.
- Configurable SLA per escalation tier (e.g., 30 min at milestone owner,
  60 min at project owner).
- `escalated_at` timestamp column on `hive.clarifications` for observability.
- Auto-escalation events logged in audit trail.

**Why deferred:** The routing chain assumes organizational depth (milestone
owners, project owners) that may not exist in early deployments. Manual
escalation covers the v1 use case. Auto-escalation requires SLA policy
decisions and background job integration that add complexity without clear
benefit until clarification volume justifies it.

**Migration impact:** Add `escalated_at` column to `hive.clarifications`.
Add escalation SLA config to execution policy or a new system config table.
Background job gains a clarification sweep query. No breaking changes.

## D4: Capability Level Routing Weight

**Origin:** V1_DESIGN.md §7.5 (filter 2), §8.3 (`level` column)

**What v1.0 does instead:** Binary eligibility — `strong` and `marginal` are
both eligible, `blocked` is ineligible. `get_next_task` does not prefer
`strong` over `marginal` agents. The three levels exist in the schema but
only two states matter for routing: eligible or not.

**What v1.1 adds:**
- Routing preference: `strong` agents are preferred over `marginal` for the
  same task, all else being equal.
- Optional `min_capability_level` on tasks (default: `marginal`) allowing
  task creators to require `strong` for critical work.
- Observability: capability match quality recorded at claim time for analysis.

**Why deferred:** No v1 routing logic differentiates `strong` from `marginal`.
Adding preference-based routing requires tie-breaking logic in `get_next_task`
that increases query complexity. The distinction becomes valuable when the
system has enough agents that routing quality matters — until then, binary
eligibility is sufficient and simpler to reason about.

**Migration impact:** None for schema. Application-only change to
`get_next_task` query ordering and optional `min_capability_level` column
on tasks.

## D5: Agent Performance Scoring and Capability Feedback Loop

**Origin:** Comparative analysis — competitors lack this, but neither does the-hive yet

**What v1.0 does instead:** Review verdicts (`approved` / `changes_requested`)
are recorded as evidence artifacts. Capability levels (`strong` / `marginal` /
`blocked`) are set manually via the capability matrix seed. There is no
automated connection between review outcomes and capability ratings.

**What v1.1 adds:**

- New table `hive.agent_performance_log` recording per-task outcomes by
  capability:

```sql
CREATE TABLE hive.agent_performance_log (
    id          SERIAL PRIMARY KEY,
    profile_id  text NOT NULL REFERENCES hive.agent_profiles(id),
    task_id     int NOT NULL REFERENCES hive.tasks(id),
    capability  text NOT NULL REFERENCES hive.agent_capabilities(id),
    outcome     text NOT NULL CHECK (outcome IN ('success', 'revision_needed', 'failed')),
    detail      jsonb NOT NULL DEFAULT '{}'::jsonb,
    created_at  timestamptz NOT NULL DEFAULT now()
);
```

- A `record_task_outcome` tool (or automatic logging on review verdict) that
  populates the log when a review task completes.
- A `get_agent_performance` tool for managers to query outcome history by
  profile and capability.
- Optional: a background job or manager tool that reads the performance log
  and recommends capability level adjustments via `update_agent_profile`.

**Why deferred:** The scoring pipeline requires decisions about aggregation
windows (last N tasks? time-weighted?), thresholds for level changes, and
whether adjustments are automatic or manager-approved. These are policy
decisions that benefit from observing real agent performance data first. v1.0
collects the raw inputs (review verdicts, evidence artifacts) — v1.1 adds the
feedback loop.

**Migration impact:** One new table, one or two new tools. No changes to
existing tables or APIs. The performance log reads from existing review
evidence and capability tables.

## D6: Shared Knowledge Base

**Origin:** Comparative analysis — agents rediscover patterns across tasks;
manager-authored context in `task_spec` is the only knowledge transfer mechanism

**What v1.0 does instead:** Knowledge lives in repo-level files (`AGENTS.md`,
`AI.md`, design docs) and in per-task fields (`context`, `task_spec`,
`acceptance_criteria`). Managers manually encode relevant knowledge into task
specs when creating work. Agents read repo files at task start. There is no
queryable, cross-task knowledge store.

**What v1.1 adds:**

- New table `hive.knowledge_entries` for operational knowledge discovered
  during execution:

```sql
CREATE TABLE hive.knowledge_entries (
    id          SERIAL PRIMARY KEY,
    project_id  int REFERENCES hive.projects(id),
    milestone_id int REFERENCES hive.milestones(id),
    domain      text,
    capability  text,
    title       text NOT NULL,
    content     text NOT NULL
                CHECK (length(content) <= 8192),
    status      text NOT NULL DEFAULT 'proposed'
                CHECK (status IN ('proposed', 'approved', 'retired')),
    proposed_by text NOT NULL,
    approved_by text,
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX ON hive.knowledge_entries (project_id, domain, status);
CREATE INDEX ON hive.knowledge_entries (capability, status);
```

- `propose_knowledge` tool — agents submit knowledge entries with status
  `proposed`. Examples: "this codebase uses factory patterns for all service
  instantiation", "module X tests take 5+ minutes — run selectively".
- `approve_knowledge` tool — managers promote `proposed` to `approved`, or
  `retire` outdated entries.
- `list_knowledge` tool — agents query at task start, filtered by project,
  milestone, domain, and/or capability. Only `approved` entries are returned
  by default.

**Design decisions:**

- **Manager-approved, not auto-populated.** Agents propose, managers approve.
  This prevents low-quality observations from polluting the knowledge base.
  The approval step is lightweight — a manager reviews proposed entries and
  promotes or discards them.
- **Repo files remain primary for stable conventions.** `AGENTS.md`, `AI.md`,
  and project docs are version-controlled and PR-reviewed. Knowledge entries
  are for operational insights that are too granular or transient for repo
  docs.
- **Vendor-neutral by design.** Knowledge entries are stored in postgres and
  retrieved via MCP tools. No vendor-specific file format. Any agent that can
  call `list_knowledge` gets the same entries regardless of vendor.
- **No vector search (pgvector) in v1.1.** Tag-based filtering (project,
  domain, capability) is sufficient for hundreds of entries. Semantic search
  via pgvector is deferred to v2+ when entry volume justifies the operational
  cost of embedding generation and index maintenance.

**Why deferred:** The knowledge base requires curation workflow (propose/approve)
and query surface (`list_knowledge` with filters) that add tools and a table
without benefiting v1.0's core goal of proving agent coordination works. v1.0
agents operate effectively with repo-level docs and manager-authored task specs.
The knowledge base becomes valuable when the system has enough task history that
patterns emerge worth capturing.

**Migration impact:** One new table, three new tools. No changes to existing
tables or APIs. Optional integration: `get_next_task` or `claim_task` response
could include relevant knowledge entries for the claimed task's domain/capability,
but this is a convenience feature, not a requirement.

## Summary

| ID | Feature | v1.0 Behavior | Additive | Breaking |
|----|---------|--------------|----------|----------|
| D1 | Task-level heartbeat | Session-derived liveness | Yes | No |
| D2 | Multi-tier review | Single `review` task type | Yes | No |
| D3 | Auto-escalation | Manual `update_clarification` | Yes | No |
| D4 | Capability routing weight | Binary eligible/blocked | Yes | No |
| D5 | Performance scoring | Manual capability ratings | Yes | No |
| D6 | Shared knowledge base | Repo files + task_spec context | Yes | No |

All six items are purely additive. v1.0 APIs and schemas remain valid after
v1.1 changes are applied.
